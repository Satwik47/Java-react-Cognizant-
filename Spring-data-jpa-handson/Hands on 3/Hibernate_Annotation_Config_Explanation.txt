Hibernate Annotation Config Implementation Walkthrough

1. Object-to-relational mapping in the persistence class `Employee`

- `@Entity`
  - Marks the `Employee` class as a persistent entity managed by Hibernate.
  - Hibernate will map instances of this class to rows in a database table.

- `@Table(name = "EMPLOYEE")`
  - Optional annotation that defines the database table name.
  - If omitted, Hibernate uses the class name by default.

- `@Id`
  - Identifies the primary key field for the entity.
  - Every persistent entity requires one.

- `@GeneratedValue(strategy = GenerationType.IDENTITY)`
  - Instructs Hibernate to let the database generate a primary key value.
  - Common strategies include `IDENTITY`, `AUTO`, `SEQUENCE`, and `TABLE`.

- `@Column(name = "FIRST_NAME")`, `@Column(name = "LAST_NAME")`, `@Column(name = "EMAIL")`
  - Maps Java fields to specific database columns.
  - Allows customizing column names, lengths, nullability, and other settings.

Example shape of the mapped entity class:

public class Employee {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private int id;

    @Column(name = "FIRST_NAME")
    private String firstName;

    @Column(name = "LAST_NAME")
    private String lastName;

    @Column(name = "EMAIL")
    private String email;

    // constructors, getters, setters
}

How mapping works:
- Each `Employee` object corresponds to one row in the `EMPLOYEE` table.
- Field values are stored in columns named after the `@Column` annotation values.
- The `id` field is the primary key, so Hibernate uses it to track entity identity.

2. End-to-end operations in Hibernate

A typical annotated Hibernate application has these core pieces:

- `hibernate.cfg.xml`
  - Contains the database connection settings, dialect, and mapping declarations.
  - Example entries:
    - JDBC URL, username, password
    - Driver class
    - `hibernate.dialect`
    - `hbm2ddl.auto` for schema generation

- `Employee` entity class
  - Defines the persistent data model with annotations.

- `SessionFactory`
  - Created once at application startup.
  - Builds `Session` objects and caches metadata.
  - Example: `SessionFactory factory = new Configuration().configure().addAnnotatedClass(Employee.class).buildSessionFactory();`

- `Session`
  - Represents a single unit of work with the database.
  - Used to begin transactions, save objects, query data, and close resources.

- `Transaction`
  - Wraps a group of database operations.
  - Example: `Transaction tx = session.beginTransaction();` and `tx.commit();`

- CRUD operations
  - Create: `session.save(employee);`
  - Read: `Employee emp = session.get(Employee.class, id);`
  - Update: modify entity fields and commit transaction
  - Delete: `session.delete(employee);`

3. Typical flow in the sample application

- Build `SessionFactory` from configuration.
- Open a `Session`.
- Begin a `Transaction`.
- Create an `Employee` object and call `session.save(employee)`.
- Commit the transaction.
- Close the session and factory.

4. Output and behavior

- When `session.save()` is called, Hibernate generates an `INSERT` SQL statement.
- The transaction commit flushes the session to the database.
- Hibernate logs SQL and transaction events to the console if logging is enabled.

5. Example of how operations map to SQL

- `session.save(employee)` -> `INSERT INTO EMPLOYEE (FIRST_NAME, LAST_NAME, EMAIL) VALUES (?, ?, ?)`
- `session.get(Employee.class, 1)` -> `SELECT * FROM EMPLOYEE WHERE id = ?`

6. End-to-end implementation aspects

- Configuration:
  - The `hibernate.cfg.xml` file and annotated entity must be consistent.
  - The entity must be registered with `Configuration` (either via XML mapping or `addAnnotatedClass`).

- Entity lifecycle:
  - Transient: newly created object not yet saved.
  - Persistent: object associated with a session and tracked by Hibernate.
  - Detached: object previously persisted but no longer associated with an open session.

- Transaction handling:
  - Always use transactions for save, update, delete operations.
  - Read-only queries can often be executed without explicit transactions, but best practice is to use them.

- Error handling:
  - Roll back the transaction if any exception occurs during database operations.
  - Close session and factory in finally blocks to avoid resource leaks.

7. What `output.txt` typically contains

- SQL statements generated by Hibernate
- Confirmation messages like `Employee saved successfully`
- Retrieved entity data from queries
